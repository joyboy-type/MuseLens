# MuseLens 指南维护与证据规则

## 1. 事实来源优先级

发生冲突时按以下顺序处理：

1. 当前版本对应的机器可读 `artifacts/evaluations/*.json`；
2. `CHANGELOG.md`、Git tag、`pyproject.toml` 和实际配置/代码；
3. 同版本专项结果文档；
4. README、作品集和面试稿；
5. 早期计划、学习日志和研究笔记。

早期 `PROJECT_PLAN.md`、`ENGINEERING_AUDIT.md` 等可能保留当时的“暂不实施”，不能据此否定
v0.1.0 后来完成的 Docker、部署、近似重复等功能。

## 2. 指标最小字段

每次写指标必须同时记录：

- 任务与相关性定义；
- 数据来源、候选集大小、查询数和语言；
- 指标名称及 K；
- 模型、索引/精排配置；
- 运行环境和日期（延迟/内存尤其需要）；
- 机器可读 artifact 路径；
- 失败案例与适用边界。

不得只写“准确率”“速度提升”或把 Recall@K、Hit@K、拒绝率混成一个数字。

## 3. 新版本更新清单

发布 `vX.Y.Z` 时：

1. 复制/更新评测 artifact，文件名含协议或版本，不覆盖历史结果；
2. 更新本目录 README 的基线版本；
3. 更新项目说明书的范围、关键结果、故障和限制；
4. 更新系统设计的组件、状态与部署边界；
5. 更新模型卡的数据、模型、阈值与未评估风险；
6. 更新面试稿，但每个口述数字必须能回链 artifact；
7. 检查 README、CHANGELOG 与本指南是否互相矛盾；
8. 实际运行测试后才记录“通过”，同时记录 commit 和日期；
9. 检查演示 URL、许可和署名仍有效。

## 4. 变更记录模板

```markdown
## YYYY-MM-DD / vX.Y.Z

- 变更：
- 原因：
- 影响的文档：
- 新证据：
- 旧指标是否仍可比：
- 新限制或风险：
```

## 5. 面试前 10 分钟检查

- 演示 URL 能打开，若不能则改用 GIF/本地 Docker，不隐瞒；
- GitHub tag、README 与简历版本一致；
- 准备一个成功案例、一个失败案例、一个未上线决策；
- 能解释 R@K、Hit@K、余弦相似度和概率的区别；
- 能指出 100k 是合成向量内存测试，不是 100k 真实图库验收；
- 不背诵超过三个数字，每个数字都带协议。
