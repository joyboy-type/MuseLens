# MuseLens v0.1.0 项目说明书

## 1. 项目摘要

MuseLens 解决的是“如何在不依赖文件名和第三方图库 API 的情况下搜索自己的图片”。用户可
导入本地图片，用中文、英文或另一张图片检索，并进行去重、标签纠正和相册整理。公开演示
额外提供固定许可图库和访客临时图库，用真实上传—索引—检索—删除链路证明系统不是静态
页面或关键词映射。

- 仓库：<https://github.com/joyboy-type/MuseLens>
- 演示：<https://sinbaby-muselens.ms.show>
- 版本：`v0.1.0`
- 许可证：MIT；公开演示图片的单独署名见 `demo_assets/ATTRIBUTIONS.md`
- 技术栈：React、TypeScript、Vite、FastAPI、PyTorch、Transformers、SigLIP2、SQLite、
  NumPy mmap、Docker、GitHub Actions、ModelScope

## 2. 需求与范围

### 核心用户任务

1. 导入图片且不移动、覆盖或删除原始照片；
2. 用自然语言或图片找到语义相似内容；
3. 看到导入/索引进度，并可从失败任务重试；
4. 发现重复图片、修正自动标签、组织智能和自定义相册；
5. 在公开演示中安全验证自己的图片，且不同访客互不可见。

### 非目标

- v0.1.0 不是多人云相册，没有账户、跨设备同步或长期云存储；
- 没有承诺百万图、分布式、高并发或强一致生产 SLA；
- 免费 CPU 演示不默认启用 2B 参数 Qwen3-VL 精排器；
- 余弦相似度不是概率，轻量模式不能可靠判断图库外查询。

## 3. 从问题到方案

普通文件搜索依赖文件名/元数据；MuseLens 用双塔模型把图片和文本编码到同一 768 维空间，
对 L2 归一化向量做点积（等价于余弦相似度）并排序。元数据筛选与语义召回在服务端组合，
结果通过同源 FastAPI 返回 React 页面。

系统有两种能力边界：

- `local`：持久化个人图库，支持导入、任务、标签、相册和重启恢复；
- `demo`：固定图库服务端强制只读；访客上传进入随机会话的独立目录、SQLite 和索引，受
  数量、像素、文件大小、总容量、活动会话数和 30 分钟 TTL 限制。

## 4. 可核验结果

以下结果不可脱离各自协议横向比较：

| 问题 | 协议 | v0.1.0 结果 | 证据 |
| --- | --- | ---: | --- |
| 本地端到端文本搜图 | 100 图，Flickr8k 500 英文查询 | R@1 91.6%，P95 20.51 ms | `local-library-e2e-100-v1.json` |
| 中英文配对检索 | 同一 100 图，各 30 条查询 | 英文 R@1 100%，中文 96.67% | 同上 |
| COCO 规模 HTTP 检索 | 5,000 图，5,000 次请求 | R@10 77.82%，均值 18.31 ms | `coco-val2017-live-api-v1.json` |
| 以图搜图鲁棒性 | 500 图，2,500 张五类扰动图 | R@1 99.36%，R@5 99.96% | `image-retrieval-coco500-v1.json` |
| 精确索引优化 | 5,000 图，1,000 查询 × 5 轮 | NumPy 比逐向量快 10.87×，Top-10 一致 100% | `vector-index-5k-v1.json` |
| 内存优化 | 100,000 × 768 合成 float32 向量 | 搜索后 RSS 679.8 → 74.7 MB（-89%） | `vector-index-memory-100k-v1.json` |
| 线上轻量检索 | 24 图，84 条双语正例 | Hit@1 72.62%，Hit@5 95.24% | `modelscope-live-bilingual-v2.json` |
| 线上临时图库 | 3 图，6 条中英文查询 | 6/6 Top-1；隔离、私有缓存、清理通过 | `modelscope-live-temporary-gallery-v2.json` |

训练轻量 Adapter 的结论同样重要：5,000/500/100 图 train/validation/test 协议下，英文
R@1 与冻结基线同为 91.6%，中文从 96.67% 降至 93.33%，因此决策为 `do_not_deploy`。

## 5. 关键工程决策

### 决策 A：冻结 SigLIP2 上线，不上线自训 Adapter

- 背景：作品集需要展示训练能力，但产品需要独立测试收益；
- 选择：保留训练代码、权重、消融和负结果，生产继续用冻结基线；
- 后果：少一项“模型已微调”的宣传点，但避免无收益权重和中文回退。

### 决策 B：默认 mmap 精确检索，保留 NumPy / FAISS 接口

- 背景：当前规模不需要 ANN，但完整向量矩阵常驻内存会扩大 RSS；
- 选择：连续 float32 mmap 缓存、分块精确点积；SQLite 为数据源，缓存可重建；
- 后果：排名可解释且内存更低，但查询仍为 O(N)，大规模时需重新基准。

### 决策 C：单容器、同源部署

- 背景：SPA 不需要 SSR，双服务会增加 CORS、代理与运维面；
- 选择：Vite 构建静态资源，FastAPI 同时提供页面、API、图片与推理；
- 后果：部署简单；模型服务和 API 不能独立扩缩容，适合当前单实例演示。

### 决策 D：公开演示分固定图库与临时会话

- 背景：固定页面不能证明完整能力，直接开放个人图库 API 又不安全；
- 选择：固定图库只读，临时图库会话隔离、限额、串行编码、TTL 和主动删除；
- 后果：可以验证真实链路，但会话 ID 只是临时访问凭据，不等于认证。

## 6. 真实故障、修复与教训

### ModelScope 运行时无法访问 Hugging Face

模型权重若在首次请求时下载，会导致冷启动失败或长期不可用。最终将公开权重放到 Docker
构建阶段缓存，并在运行时离线加载。教训是：健康检查必须覆盖“模型已加载 + 固定索引可用”，
不能只测首页 200。

### “上传成功”但搜索链路未必成功

单测上传接口无法覆盖后台任务、会话路由、索引可见性和清理。项目增加真实临时图库部署门：
创建会话、上传 dog/car/pizza、等待索引、双语 Top-1 检索、验证未知会话 404、缓存
`private, no-store`，最后删除并确认资源消失。

### FAISS 更快但本地运行栈有冲突

FAISS `IndexFlatIP` 在 5,000 图纯索引基准为 0.237 ms，快于 NumPy 的 0.363 ms；但项目
记录了 macOS ARM64 下 PyTorch 与 FAISS wheel 的 OpenMP 冲突。因当前端到端瓶颈不只在
纯索引，默认选择依赖更少的 mmap，并保留可选后端，而非为较小收益扩大安装风险。

### 早期绝对阈值造成错误拒绝

SigLIP2 正例与图库外负例的最高分区间重叠，单一余弦阈值不能可靠拒答。轻量演示改为相对
排序且不把分数称为概率；严格拒答仅在有独立评测的 Qwen3-VL 精排模式中启用。

## 7. 测试与交付

仓库当前有 18 个 Python 测试文件、81 个静态扫描可见的 `test_*` 函数；该数量不是本次运行
通过数。测试覆盖 API、索引、会话、任务、重复检测、标签、检索筛选、评估、精排和部署脚本。
前端另有智能相册、相关性和构建产物测试。CI 执行 Ruff、Pytest、前端 lint/build/test，
生产发布后再运行双语与临时图库合同。

## 8. 已知限制与下一步

最值得做的下一步不是继续堆模型，而是：

1. 用真实 50k+ 图库比较 mmap、HNSW/IVF 和独立向量服务的召回—P95—内存；
2. 扩大并版本化中英文、长尾、小目标和图库外查询集；
3. 若要多人使用，引入真实认证、对象存储、速率限制和数据删除审计；
4. 为线上服务增加持续的延迟、失败率、内存和模型/数据版本观测；
5. 录制 60 秒演示，但保持机器可读评测为主要证据。
