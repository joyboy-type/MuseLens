# MuseLens 面试讲解稿

这份文档用于简历描述、项目答辩和技术面试。所有数字都来自仓库内可复现的实验，不要背诵未验证的结论。

## 30 秒项目介绍

> MuseLens 是我实现的本地优先多模态图片检索与整理系统。用户可以导入自己的图库，再用中文、英文或另一张图片搜索。系统不是调用第三方图库 API，而是用 SigLIP2 生成图文向量，用 SQLite 保存元数据、标签和相册关系，并通过低内存 mmap 索引完成检索。我还实现了可人工纠正的自动标签、智能/自定义相册、去重、后台导入和临时会话隔离。公开版本部署在 ModelScope，访客可以上传自己的图片验证完整链路。

## 两分钟展开

可以按“问题—方案—证据—取舍”四步讲：

1. **问题**：系统既要让普通用户搜索自己的图片，也要证明后端真的完成了图文检索，而不是一个固定展示页面。
2. **方案**：React/TypeScript 负责交互，FastAPI 负责导入、任务和搜索；SigLIP2 把图片和中英文文本编码到同一向量空间；SQLite 和 NumPy 分别处理持久化及精确检索。
3. **证据**：100 张本地图库的 500 条英文查询 Recall@1 为 91.6%；2,500 张扰动图的以图搜图 Recall@1 为 99.36%；5,000 图真实 HTTP 查询平均延迟从 51.40 ms 优化到 18.31 ms；线上 84 条自然双语查询 Hit@5 为 95.24%。
4. **取舍**：我训练了轻量 Adapter，但独立测试集没有优于冻结基线，所以没有上线；FAISS 纯索引更快，但在当前 M4/PyTorch 环境存在运行时冲突，因此先用 NumPy 验证精确基线，再把默认后端升级为排名一致的低内存 mmap 索引。

## 最值得讲的三个难点

### 1. 防止“能索引但搜不到”的假完成

- 症状：上传接口成功，并不代表后续文本搜索、会话路由和结果展示都正确。
- 处理：建立端到端契约，真实创建临时会话，上传 dog/car/pizza 图片，等待后台索引，再用中英文逐一验证 Top-1。
- 进一步验证：检查未知会话返回 404、图片响应为 `private, no-store`、删除后再次访问返回 404。
- 价值：CI 验证的是用户链路，不只是单元函数或首页状态码。

### 2. 在准确率、延迟和稳定性之间做索引选择

- 旧实现逐向量执行 Python 点积，5,000 图平均 3.942 ms。
- 将向量堆叠为连续 `float32` 矩阵后，单次矩阵乘法降到 0.363 ms，加速 10.87 倍。
- FAISS `IndexFlatIP` 达到 0.237 ms，但 macOS ARM64 的 PyTorch 与 FAISS wheel 存在 OpenMP 冲突。
- 最终决策：保留统一索引接口和可选 NumPy / FAISS，默认使用 mmap；10 万向量搜索后 RSS 约降低 89%，Top-10 排名保持一致。

### 3. 把个人本地工具安全地变成公开演示

- 本地模式需要持久化导入；公开模式不能允许陌生人修改官方图库。
- 服务端按运行模式强制权限，而不是仅在前端隐藏按钮。
- 公开访客图库使用随机会话、独立目录和索引，限制文件数量、单文件大小、总容量和活动会话数。
- 会话支持主动清除，并在索引完成 30 分钟后过期；服务重启不会恢复访客数据。
- GitHub Actions 只组装最小运行包并推送到 ModelScope，部署后自动运行质量门；包未改变时跳过约 30 分钟的重建。

## 高频追问

### 这和普通关键词搜索有什么不同？

文件名中不需要出现查询词。图片和文本由双塔模型映射到同一语义空间，例如一张文件名为 UUID 的汽车图片仍可以被“汽车”或 `a red car` 检索。系统也会明确区分语义相似度与概率，避免把余弦分数伪装成置信度。

### 为什么选择 SigLIP2？

项目先做了模型对比。CLIP ViT-B/32 在这组中文配对查询上的 Recall@1 只有 3.3%，SigLIP2 提升到 96.7%，英文保持 100%。因此选择来自同一协议下的实测，而不是模型热度。

### 你不是训练了模型吗，为什么最终没用？

训练不是目的，独立测试集收益才是。冻结主干的残差 Adapter 在 5,000 张训练图上完成多组 batch size 和学习率消融，但英文收益近乎为零，中文 Recall@1 下降 3.4 个百分点。继续上线会增加权重管理和过拟合风险，所以保留训练代码、权重与负结果，生产继续使用基线。

### 数据扩大到 10 万张怎么办？

当前 mmap 精确索引仍会线性扫描，只是避免让完整向量矩阵常驻进程内存。后续先用相同协议测量内存、P95 和召回；当数据证明需要时，再将统一 `VectorIndex` 接口替换为 HNSW/IVF 或独立向量服务，并补充分页、增量建索引和 ANN 召回损失评估。不会只因为“工业界常用”就提前引入运维复杂度。

### 如何保证项目不是只在你的电脑上工作？

后端、前端、Docker 构建和线上契约都进入 GitHub Actions。生产使用与本地同一 FastAPI 应用和 Dockerfile，ModelScope 线上健康状态要求模型已加载、24 张固定图片可用、固定图库只读且临时图库开启。

### 当前局限是什么？

- 免费 CPU 演示默认不运行 2B 参数 VL 精排器，所以固定演示图库的轻量召回准确率低于本地高精度模式。
- 当前没有账户系统，临时图库不适合长期或敏感数据。
- 免费轻量模型能排序，但 10 条图库外查询的拒答率为 0%；正负最高分区间重叠，不能把余弦分数伪装成可靠置信度，严格拒答需要可选 VL 精排器。
- 5,000 图验证了单机精确索引，但还不能声称支持百万级图库。
- 公开演示数据规模较小，不能用其结果代替 COCO 和 Flickr8k 的独立评测。

## 简历描述示例

不要一次塞入全部指标，可按岗位选择 2–3 条：

- 独立实现本地优先的多模态图片搜索系统，使用 React/TypeScript、FastAPI、PyTorch、SigLIP2 与 SQLite，支持中英文文本搜图、以图搜图、后台批量导入、去重及重启恢复。
- 将 5,000 图精确向量检索从 Python 逐向量计算优化为 NumPy 连续矩阵，纯索引平均延迟降低 90.8%（3.942 ms → 0.363 ms），Top-10 排名保持 100% 一致。
- 设计公开访客临时图库的配额、会话隔离和 TTL 清理机制，并在 GitHub Actions 中建立真实上传—索引—双语检索—删除的部署质量门。
- 设计自动标签的人在回路纠正和 SQLite 多对多相册模型，批量重建保护人工结果，自定义相册仅保存引用且删除不影响原图。
- 将线上检索合同扩展为 84 条中英文自然查询，Hit@5 达 95.24%；同时识别轻量模型图库外拒答率为 0% 的边界并保留完整失败报告。
- 在 5,000 张训练图上完成轻量双塔 Adapter 消融；因独立测试未超过冻结 SigLIP2 基线而拒绝上线，以实验结果驱动模型选型。

## 代码演示顺序

面试官要求看代码时，建议只打开五处：

1. `src/muselens/api.py`：HTTP 契约和 local/demo 权限边界。
2. `src/muselens/library.py`：导入、批量编码、去重和持久化流程。
3. `src/muselens/index.py`：mmap / NumPy / FAISS 的统一索引接口与稳定排序。
4. `src/muselens/sessions.py`：临时会话隔离、配额和 TTL。
5. `.github/workflows/deploy-modelscope.yml`：从发布到真实线上验收的闭环。
